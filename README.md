# NomadeoPMS

An open-source, multi-tenant Property Management System REST API for adventure travel businesses — surf camps, yoga retreats, dive centers, and similar operators.

Built with **Laravel 11** + **PostgreSQL 16**. No frontend — pure REST API with auto-generated OpenAPI docs.

---

## Features

- **Multi-tenancy** — complete data isolation per organizer via column-based scoping; no cross-tenant data leaks
- **Multi-property** — one organizer can manage multiple properties, each with its own room types, units, and rules
- **Inventory management** — room types (dormitory, private, shared) with individual bookable units
- **Programs & add-ons** — package-based programs (surf week, yoga retreat) with attachable add-ons (board rental, transfer, meals)
- **Availability engine** — per-day calendar, date-range blocking, weekday masks, unit-level overrides
- **Booking rules** — min/max nights, check-in day restrictions, advance booking windows (min/max days)
- **Pricing engine** — season pricing, per-night and fixed-package models, priority-based rule resolution, `brick/money` for precise arithmetic
- **Discount codes** — percentage and fixed-amount discounts with validity windows and usage caps
- **Full booking lifecycle** — pending → confirmed → checked_in → checked_out, with cancellation and no-show support
- **Payment tracking** — deposit + balance schedules, payment rules per property, refund recording
- **Staff management** — invite-by-email flow with secure one-time tokens, role-based access (`organizer_admin`, `organizer_staff`)
- **Customer profiles** — with nationality, VIP status, and booking history
- **Reviews** — post-stay review with sub-ratings, moderation flag
- **Reporting** — occupancy rate, revenue, booking stats, customer stats — all filterable by property and date range
- **Marketplace integration** — hashed bearer tokens (no JWT required), per-day availability calendar, auto-unit assignment, customer upsert
- **Activity log** — immutable audit trail on bookings, customers, and payments

## Quick Start

### 1. Clone and configure

```bash
git clone https://github.com/nomadeo/pms.git
cd pms
cp .env.example .env
```

Edit `.env` — the defaults work with the Docker Compose setup out of the box. The only values you must set are:

```env
APP_KEY=          # filled by artisan below
JWT_SECRET=       # filled by artisan below
```

### 2. Start containers

```bash
docker compose up -d
```

Services started:

| Container | Role | Port |
|---|---|---|
| `pms_app` | PHP-FPM 8.4 | — |
| `pms_webserver` | Nginx | **8001** |
| `pms_db` | PostgreSQL 16 | 5433 |
| `pms_redis` | Redis 7.2 | 6380 |
| `pms_queue` | Queue worker | — |

### 3. Generate keys

```bash
docker exec pms_app php artisan key:generate
docker exec pms_app php artisan jwt:secret
```

### 4. Run migrations and seed

```bash
# Fresh database with roles, super admin, and demo organizer
docker exec pms_app php artisan migrate:fresh --seed
```

This creates:

| Account | Email              | Password | Role |
|---|--------------------|---|---|
| Super admin | `admin@nomadeo.es` | `secret` | `super_admin` |
| Organizer admin | `manager@test.com` | `secret` | `organizer_admin` |
| Staff member | `staff@test.com`   | `secret` | `organizer_staff` |

The demo organizer (`Ocean's Surf Camp`) comes with a property, room types, units, programs, add-ons, pricing rules, booking rules, and sample bookings.

### 5. Generate API docs

```bash
docker exec pms_app php artisan l5-swagger:generate
```

Open **http://localhost:8001/api/documentation** — interactive Swagger UI with every endpoint.

## API Overview

All endpoints are prefixed `/api/v1/`. Authenticated endpoints require a JWT bearer token obtained from `POST /api/v1/auth/login`.

### Auth

```
POST   /api/v1/auth/login           Login — returns JWT
GET    /api/v1/auth/me              Current user info
POST   /api/v1/auth/logout          Invalidate token
POST   /api/v1/auth/accept-invite   Accept staff invite and set password
```

### Organizer & Properties

```
GET    /api/v1/organizers           List organizers (super_admin only)
POST   /api/v1/organizers           Create organizer (super_admin only)
GET    /api/v1/properties           List properties for current organizer
POST   /api/v1/properties           Create property
```

### Inventory

```
GET    /api/v1/room-types?property_id=   List room types
POST   /api/v1/room-types                Create room type
GET    /api/v1/units?property_id=        List units
POST   /api/v1/units                     Create unit
```

### Programs & Add-Ons

```
GET    /api/v1/programs             List programs
POST   /api/v1/programs             Create program
GET    /api/v1/add-ons              List add-ons
POST   /api/v1/add-ons              Create add-on
```

### Availability & Booking Rules

```
GET    /api/v1/availability?property_id=&start_date=&end_date=
GET    /api/v1/availability/check?property_id=&check_in_date=&check_out_date=
POST   /api/v1/availability/rules
DELETE /api/v1/availability/rules/{id}
GET    /api/v1/booking-rules
POST   /api/v1/booking-rules
DELETE /api/v1/booking-rules/{id}
```

### Pricing & Discounts

```
GET    /api/v1/pricing-rules
POST   /api/v1/pricing-rules
DELETE /api/v1/pricing-rules/{id}
GET    /api/v1/discounts
POST   /api/v1/discounts
```

### Bookings

```
GET    /api/v1/bookings             List bookings (filterable)
POST   /api/v1/bookings             Create booking
GET    /api/v1/bookings/{id}        Get booking detail
PATCH  /api/v1/bookings/{id}        Update booking status
DELETE /api/v1/bookings/{id}        Cancel booking
GET    /api/v1/bookings/calendar?property_id=&month=
```

`unit_ids` is **required** when creating a booking — the API does not auto-assign units on direct bookings.

### Customers

```
GET    /api/v1/customers            List customers
POST   /api/v1/customers            Create customer
GET    /api/v1/customers/{id}       Customer detail + booking history
PATCH  /api/v1/customers/{id}       Update customer
```

### Payments

```
GET    /api/v1/payments?booking_id=     List payments for a booking
POST   /api/v1/payments                 Record a payment
GET    /api/v1/payment-rules
POST   /api/v1/payment-rules
```

### Staff

```
GET    /api/v1/staff                List staff members
POST   /api/v1/staff/invite         Invite a staff member (returns one-time token)
PATCH  /api/v1/staff/{userId}/role  Update staff role
DELETE /api/v1/staff/{userId}       Remove staff access
```

### Reporting

```
GET    /api/v1/reports/occupancy?property_id=&start_date=&end_date=
GET    /api/v1/reports/revenue?property_id=&start_date=&end_date=
GET    /api/v1/reports/bookings?property_id=&start_date=&end_date=
GET    /api/v1/reports/customers?property_id=&start_date=&end_date=
```

### Marketplace Integration

These endpoints use **hashed bearer tokens** (no JWT). Tokens are created by an organizer admin via `POST /api/v1/integration/tokens`.

```
POST   /api/v1/integration/tokens                       Create integration token (JWT auth)
GET    /api/v1/integration/{propertySlug}/availability  Per-day availability calendar
POST   /api/v1/integration/{propertySlug}/bookings      Create booking from marketplace
POST   /api/v1/integration/{propertySlug}/customers     Upsert customer
```

The availability response returns the same per-day calendar as the internal endpoint, making it straightforward to power a booking widget on an external marketplace.

## Architecture

```
app/
├── Tenant/          TenantContext singleton + TenantScope global Eloquent scope
├── Models/          All Eloquent models (flat)
├── Auth/            JWT login / logout / refresh / accept-invite
├── Organizer/       Organizer + Property CRUD
├── Inventory/       RoomType + Unit
├── Program/         Programs + AddOns
├── Availability/    AvailabilityRule + BookingRule + calendar use cases
├── Pricing/         PricingRule + Discount + price calculation
├── Customer/        Customer profiles + Reviews
├── Booking/         Full booking lifecycle — CreateBookingUseCase is the orchestrator
├── Payment/         Payment recording + PaymentRule
├── Staff/           Invite flow + role management
├── Reporting/       Occupancy, revenue, and booking analytics
└── Integration/     Marketplace integration (token auth)
```

Each domain contains `Controllers/`, `UseCases/`, `Repositories/` (interface + Eloquent implementation), and `Requests/`. All repository interfaces are bound in `AppServiceProvider`.

### Multi-tenancy

Every request that carries a JWT has its `organizer_id` claim extracted by the `ResolveTenant` middleware and placed into the `TenantContext` singleton. A global Eloquent scope (`TenantScope`) then automatically appends `WHERE organizer_id = ?` to every query on tenant-scoped models.

Super admins (`organizer_id: null` in JWT) bypass the scope globally and can target a specific organizer by passing `X-Organizer-ID: <uuid>`.

Integration routes set `TenantContext` from the `integration_tokens` table instead of a JWT.

## Staff Invite Flow

1. Admin calls `POST /api/v1/staff/invite` with `email` and `role`.
2. A user record is created with `is_active = false` and a SHA-256 hashed invite token (72-hour TTL).
3. The plain token is returned in the response **once** — send it to the user out-of-band (email, etc.).
4. The user calls `POST /api/v1/auth/accept-invite` with the token and their chosen password.
5. The account is activated, the token is cleared, and a JWT is returned immediately.

## Production Deployment

A production-optimised multi-stage Dockerfile is provided at `docker/Dockerfile.prod`. It:

- Installs only `--no-dev` Composer dependencies
- Runs `php artisan config:cache`, `route:cache`, and `view:cache`
- Starts PHP-FPM and the queue worker via `supervisord`

The Swagger UI is disabled automatically when `APP_ENV != local`.
